# Feature Specification: 得点入力エリア拡大・プレイヤー上限10人対応

**Feature Branch**: `007-layout-max-players`
**Created**: 2026-03-01
**Status**: Draft
**Input**: User description: "得点入力のボタン領域をもう少し縦に長くしたい。プレイヤー一覧領域は4人まではスクロール不要、5人以上は画面の縦サイズによってはスクロールしてもよいことにする。合わせて、プレイヤー登録上限を10まで上げる。"

## Clarifications

### Session 2026-03-01

- Q: ボタン拡大と「4人以下スクロールなし（FR-003）」が競合する場合の優先度は？ → A: 4人以下でスクロールなしを維持する範囲で、できる限りボタンを大きくする（FR-003 を上位制約とする）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 得点入力ボタンを広く使えるようにする (Priority: P1)

ゲーム中に得点を入力するとき、0〜12のボタンが現在よりも縦に大きく表示される。ボタンが大きいほど素早く正確にタップでき、屋外での立ちプレイ中でも快適に操作できる。

**Why this priority**: 得点入力はゲームの中心操作であり、ボタンが大きいほど操作ミスが減り直接ゲーム体験に影響するため最優先。

**Independent Test**: ゲームを開始し、得点入力エリアのボタン（0〜12）が以前より縦に大きく表示されていることを目視確認し、各ボタンが正常に機能することを確認する。

**Acceptance Scenarios**:

1. **Given** ゲームが進行中、**When** 得点入力エリアを表示する、**Then** 0〜12の各ボタンが以前より縦に大きく表示される
2. **Given** 得点入力エリアのボタンが拡大された状態で、**When** ボタンをタップして得点を入力する、**Then** 得点入力が正常に機能し、スコアが更新される
3. **Given** 2〜4人でゲームが進行中、**When** ゲーム画面を表示する、**Then** 拡大された得点入力ボタンとプレイヤー一覧が同時に1画面内に収まる

---

### User Story 2 - 5人以上でのプレイヤー一覧スクロール許容 (Priority: P2)

5人以上でゲームをするとき、プレイヤー一覧が画面の縦サイズに応じて自動的にスクロール可能になる。4人以下ではこれまで通りスクロール不要で全員が表示される。画面が十分に縦長であればスクロールなしで全員が見えるが、小型端末では一覧をスクロールして全員のスコアを確認できる。

**Why this priority**: US1 のボタン拡大によってプレイヤー一覧の表示領域が減るため、5人以上のレイアウト戦略を明示的に定義する必要がある。US1 と密接に関連するが独立して実装可能。

**Independent Test**: 5人・6人でゲームを開始し、画面高さが十分な端末ではスクロールなしで全員が表示され、小型端末では一覧スクロールで全員確認できることを確認する。4人以下のゲームでは一覧スクロールが不要であることも確認する。

**Acceptance Scenarios**:

1. **Given** 4人でゲームが進行中、**When** ゲーム画面を表示する、**Then** プレイヤー一覧をスクロールせずに全4人が表示される
2. **Given** 5人でゲームが進行中・画面高さが十分な端末、**When** ゲーム画面を表示する、**Then** プレイヤー一覧をスクロールせずに全5人が表示される
3. **Given** 5人でゲームが進行中・小型端末（画面が縦に短い）、**When** ゲーム画面を表示する、**Then** プレイヤー一覧をスクロールすることで全5人のスコアを確認できる
4. **Given** 5〜10人でゲームが進行中、**When** プレイヤー一覧をスクロールする、**Then** 得点入力エリアは常に画面内に表示されたまま（一覧エリア内スクロール）

---

### User Story 3 - プレイヤーを最大10人まで登録できる (Priority: P3)

ゲーム開始前のプレイヤー設定画面で、最大10人までプレイヤーを追加できる。現在の上限6人から引き上げることで、大人数のグループでも1デバイスでゲームを楽しめる。

**Why this priority**: UI・ロジック両面の変更が必要だが、ゲーム体験の改善幅（6→10人）は限定的なため P3。

**Independent Test**: プレイヤー設定画面でプレイヤーを10人まで追加し、11人目の追加ができないことを確認する。10人でゲームを開始しスコア入力が正常に動作することを確認する。

**Acceptance Scenarios**:

1. **Given** プレイヤー設定画面を開く、**When** プレイヤーを1人ずつ追加する、**Then** 10人まで追加できる
2. **Given** プレイヤーが10人登録されている、**When** さらに追加しようとする、**Then** 追加ボタンが無効になるか非表示になり11人目を追加できない
3. **Given** 10人でゲームが開始された、**When** ゲーム画面を表示する、**Then** 全10人のスコアが確認でき、得点入力が正常に機能する
4. **Given** 既存の2〜6人のゲーム設定、**When** プレイヤー設定を行う、**Then** 従来通り2〜6人でも問題なくゲームを開始できる

---

### Edge Cases

- 7〜10人のゲームで、脱落者が出てプレイヤー数が実質的に減った場合も一覧表示が正常に機能すること
- 10人ゲームで画面が極端に小さい端末（高さ 600px 未満）では表示が崩れる可能性があるが、600px 以上を最低対応とする
- 横向き（landscape）表示は既存の md: レイアウトが対応しているため対象外
- プレイヤー名の最大文字数（12文字）は変更しない

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 得点入力エリアのボタン（0〜12）の縦の高さが現在より大きく表示されること。ただし FR-003（4人以下スクロールなし）を上位制約とし、その範囲内で最大限大きくすること
- **FR-002**: 得点入力エリアは常に画面内に表示され、プレイヤー一覧によって隠れないこと
- **FR-003**: プレイヤーが4人以下のとき、プレイヤー一覧内でスクロールなしに全員が表示されること（ボタン拡大に対する上位制約）
- **FR-004**: プレイヤーが5人以上のとき、プレイヤー一覧は画面の縦サイズに応じてスクロール可能になること（得点入力エリアは常に画面内に固定）
- **FR-005**: プレイヤー設定画面で登録できるプレイヤーの上限が10人になること
- **FR-006**: 10人でゲームを開始でき、スコア計算・undo・ターン管理が正常に動作すること
- **FR-007**: 既存の横向き（md: 幅以上）レイアウトを維持すること

### Key Entities

- **プレイヤー上限**: ゲーム設定における最大プレイヤー数。現在は6人、本機能で10人に変更。
- **得点入力エリア高さ**: プレイヤー一覧と得点入力の縦スペース配分比率。本機能で得点入力側を拡大する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 得点入力ボタンの縦の高さが変更前より大きくなっていること（目視確認）。かつ4人以下ではプレイヤー一覧と得点入力が同時にスクロールなしで表示されること
- **SC-002**: 2〜4人ゲームにおいて、プレイヤー一覧と得点入力エリアがスクロールなしで同時に1画面に収まること
- **SC-003**: 5〜10人ゲームにおいて、得点入力エリアが常に画面内に表示されていること
- **SC-004**: プレイヤー設定画面で10人まで登録でき、11人目の追加が拒否されること
- **SC-005**: 10人でゲームを開始し、全員のスコア入力・undo が正常に機能すること
- **SC-006**: 既存の全ユニットテストが引き続きパスすること

## Assumptions

- 最低対応画面高さは 600px（一般的なスマートフォンの最小高さ）
- 横向き（md: ブレークポイント以上）は既存レイアウトを維持し、変更対象外
- 「得点入力ボタンをもう少し縦に長く」とは、4人以下スクロールなし（FR-003）を満たせる最大の高さまでボタンを拡大することを指す
- プレイヤー名の最大文字数（12文字）・ゲームルール（バースト・脱落・勝利）は変更しない
- Mölkkout モードのプレイヤー上限は本機能のスコープ外とする
