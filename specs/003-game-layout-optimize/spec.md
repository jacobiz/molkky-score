# Feature Specification: ゲーム画面レイアウト最適化

**Feature Branch**: `003-game-layout-optimize`
**Created**: 2026-02-28
**Status**: Draft
**Input**: User description: "レイアウト調整（ScoreBoard の高さ上限固定・ボタン拡大）を、プレイヤー数制限に伴って行いたい"

## Clarifications

### Session 2026-02-28

- Q: 少人数時（2〜5人）のスコア入力エリアの挙動 → A: ゲーム開始時の人数に応じて伸縮（少ないほどボタンエリアが広がる）。ゲーム中の脱落ではレイアウトは変動しない。

## Context

プレイヤー数が最大6人に制限されたことで、ゲーム画面の縦方向スペースが予測可能になった。
この制約を活かし、スコアボードの表示領域を固定してスコア入力ボタンが常に画面内に収まるようにし、
ボタンを大きくして誤タップを減らす。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - スコア入力ボタンが常に画面に収まる (Priority: P1)

プレイヤーがゲーム中にスコアを入力するとき、プレイヤー数（2〜6人）にかかわらず、
スコア入力ボタンがスクロールなしで常に画面内に表示される。

**Why this priority**: スコア入力がゲームの中心操作であり、ボタンが見えない・押せない状態は
プレイ体験を大きく損なう。P1 で解決すべき最重要課題。

**Independent Test**: 2人〜6人でゲームを開始し、スコア入力ボタン（0〜12）がスクロールなしで
すべて画面内に表示されていることを目視確認できる。

**Acceptance Scenarios**:

1. **Given** 6人のゲームが進行中, **When** ゲーム画面を開く, **Then** スコア入力ボタン全体が画面内に表示され、スクロール不要で操作できる
2. **Given** 2人のゲームが進行中, **When** ゲーム画面を開く, **Then** スコア入力ボタンが同様に表示され、余白が生じても操作に問題がない
3. **Given** プレイヤーが脱落して人数が減った場合, **When** スコア入力ボタンを操作する, **Then** ボタンが常に画面内に収まっている

---

### User Story 2 - スコア入力ボタンが大きく押しやすい (Priority: P2)

プレイヤーが屋外・立った姿勢でスマートフォンを操作する状況でも、スコア入力ボタンを
確実に1タップで押せる十分なサイズになっている。

**Why this priority**: P1（ボタンが見える）の前提の上で、誤タップ防止・快適操作のために重要。
US1 完了後に独立して改善できる。

**Independent Test**: ゲーム画面でスコア入力ボタン（0〜12）の各ボタンの高さが
タッチターゲット推奨サイズ（44px 以上）を満たしていることを確認できる。

**Acceptance Scenarios**:

1. **Given** ゲーム画面が表示されている, **When** スコア入力エリアを確認する, **Then** 全ボタンの高さが 44px 以上である
2. **Given** 6人のゲームでスコアボードが最大表示されている, **When** ボタンサイズを確認する, **Then** US1（ボタンが画面内に収まる）を維持しながらボタンが可能な限り大きい
3. **Given** Mölkkout 画面が表示されている, **When** スコア入力エリアを確認する, **Then** 通常ゲームと同様にボタンサイズが改善されている

---

### Edge Cases

- 6人中複数名が脱落して表示行が減った場合、ボタンエリアの高さは変動しない（安定したレイアウト）
- 画面の高さが極端に小さいデバイス（例: 568px 以下）でも、スコアボードがスクロール可能になりボタンが操作できる
- Mölkkout 画面でも同様のレイアウト改善が適用される

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: スコアボード表示エリアは高さの上限を持ち、最大6行分の内容が収まる範囲で固定される
- **FR-002**: スコアボードの行数がエリアを超える場合、スコアボード内でスクロール可能になる
- **FR-003**: スコア入力ボタンエリアは常にビューポート内に表示され、スクロールなしで操作できる
- **FR-003a**: ゲーム開始時のプレイヤー数が少ないほどスコア入力エリアが広くなり、ボタンが大きくなる。ゲーム中に脱落者が出てもレイアウトは変動しない
- **FR-004**: 全スコア入力ボタン（0〜12）の高さは 44px 以上を維持する
- **FR-005**: Mölkkout 画面のスコア入力ボタンにも同じサイズ基準が適用される
- **FR-006**: タブレット・デスクトップの横並びレイアウトでは既存の動作を維持する

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 2〜6人のすべての構成でゲーム画面を開いたとき、スクロールなしにスコア入力ボタン全体が視認・操作できる
- **SC-002**: 全スコア入力ボタン（0〜12 の計13個）の高さが 44px 以上である
- **SC-003**: 6人構成でスコアボードとスコア入力エリアの合計高さが、一般的なスマートフォン画面（667px 以上）内に収まる
- **SC-004**: Mölkkout 画面においても SC-001〜SC-002 と同等の基準を満たす

## Assumptions

- プレイヤー上限は 6人（別タスクで実装済み）
- モバイルファースト（縦向き）が主要ユースケース
- ヘッダー高さは約 60px として設計
- タブレット以上（md: ブレークポイント）では既存の横並びレイアウトが引き続き機能する
- Mölkkout 画面のスコア入力 UI は通常ゲームと同一コンポーネント構造である
