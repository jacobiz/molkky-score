## Critical — これがないと大会で使えない

### 1. 投球ごとのショットクロック（Per-throw Shot Clock）

**根拠（公式ルール §11）**: 大会では 1 投球あたり 30〜45 秒の制限。超過はファウル（ミス記録）。

**現状のギャップ**: 現在の「時間切れ途中決着」はゲーム全体の打ち切りであり、投球単位のタイマーではない。審判が手動で秒読みするしかなく、公平性に欠ける。

**必要な仕様**:

- セットアップ画面で「ショットクロック: OFF / 30秒 / 45秒」を設定
- 投球入力直前から自動カウントダウン（大きな数字表示）
- 残り 5 秒で赤表示＋バイブレーション
- タイムアップ時: 確認なしで即時ミス（0点）を自動記録
- カウントダウン中でも審判が任意タイミングで得点を入力できること（ピンが倒れた瞬間に入力開始する運用）

---

### 2. マッチ管理（Best-of-N Match Series）

**根拠（公式ルール §11）**: 大会は通常 Best-of-3 形式。各ゲーム後、**負けチームが次ゲームの先攻/後攻を選択**。

**現状のギャップ**: 現在の「再戦」は同一投球順でゲームを再開するだけ。大会では:

- 各ゲームの勝敗を記録し、マッチ勝者を自動判定する必要がある
- ゲーム間で「どちらが先に投げるか」を負けたチームが選ぶ規則がある
- これを手動で管理すると混乱の元になる

**必要なフロー**:

```
マッチセットアップ:
  - ベスト of: 1 / 3 / 5（デフォルト3）
  - マッチ参加チーム/プレイヤー名登録

ゲーム間フロー:
  Game 1終了 → 勝敗表示 → 「負け側が先攻/後攻を選択」UI → Game 2開始
  ...マッチ勝者確定まで繰り返し

マッチ結果画面:
  - ゲームごとのスコア記録（スコアシートへのリンク）
  - マッチ勝者表示
  - Molkkout 発動ボタン（引き分け時）
```

---

### 3. ファウル（投球ライン越え）の明示的記録

**根拠（公式ルール §4, §7, §11）**: 足が投球ラインを越えた場合はファウル = ミス扱い。大会では厳格に適用され、公式スコアシートへの記録が必要。

**現状のギャップ**: 現在は「0点」入力のみ。「ミス（0本）」と「ファウル（ライン越え）」は区別されない。

**必要な仕様**:

- PinInput に「ファウル」ボタンを追加（0点ミスと同じ効果だが記録上は別）
- `HistoryTurnEntry` に `isFoul: boolean` フィールドを追加
- スコアシートでファウル行に「F」バッジを表示
- 統計集計時にミスとファウルを区別してカウント

## Important — 大会品質を大きく向上させる

### 4. 37点ルール（The 37-Point Rule）

**根拠（公式ルール §7, §12）**: 一部の大会で採用。ちょうど37点に3ターン連続で留まったプレイヤーは脱落。

**現状のギャップ**: 未実装。スコアが37点に留まり続けてもゲームは続く。

**実装の考慮点**: 「37点に留まった」の判定は「バストなしで37→何か→37になった回数」ではなく「37点のまま投球した連続回数」のカウントが必要。`consecutiveMisses` と同様のカウンタを `stuck37Count` として `Player` 型に追加する形が自然。

**必要な仕様**:

- セットアップ画面で「37点ルール: ON/OFF」トグル（デフォルト OFF）
- ScoreBoard で37点プレイヤーを黄色ハイライト＋残り投球回数表示
- 3回連続で37点確定時: 脱落トースト表示（`consecutiveMisses` と同一パターン）

---

### 5. チーム内投球順のトラッキング

**根拠（公式ルール §9）**: チームプレイでは「チーム内の固定投球順」が必要。

**現状のギャップ**: 現在の実装でチーム名を1プレイヤーとして登録すれば得点共有は動作するが、審判がどのチームメンバーが投げる番か管理できない。

**実装アプローチ**: 「チーム名」と「メンバー名の配列」を持つ構造を `SetupScreen` に追加。チームの得点は1つだが、現在投球すべきメンバーをローテーションで画面に表示する。

```typescript
// 将来の型イメージ
interface TeamSetup {
  teamName: string
  members: string[]  // 投球順、1名でも可（個人プレイと統一）
}
```

---

### 6. スコアシートの印刷・PDF エクスポート

**根拠**: 大会では公式記録として紙への印刷が求められることが多い。

**現状のギャップ**: `ScoreSheetModal` はスクリーン表示のみ。`@media print` スタイルなし。

**必要な仕様**:

- ScoreSheetModal に「印刷」ボタン追加
- `@media print` CSS: モーダルのみフルページで印刷、操作ボタン類を非表示
- ヘッダーに試合日時・参加者・勝者を明記

---

### 7. プレイヤー通算成績・統計

**根拠**: 大会のシード順や実力評価に使う。

**現状のギャップ**: `HistoryScreen` は過去のゲーム一覧を参照できるが、プレイヤー横断の集計はない。

**必要な集計項目**:

- 勝率・ゲーム数
- 平均最終スコア
- バースト回数・脱落回数・ファウル回数
- Molkkout 戦績

**実装**: `HistoryScreen` 内に「統計」タブを追加し、`loadHistory()` から集計する。

## Nice-to-have — 大会体験を洗練する

### 8. 観戦モード（Read-only Spectator View）

大型スクリーンやサブデバイスで現在の試合スコアを表示専用で参照できるビュー。
`localStorage` の変更を `storage` イベントでリッスンすれば、同一デバイスの別タブでリアルタイム同期可能。

### 9. QRコードによるデータ共有

試合結果（`GameHistoryRecord`）をQRコードにエンコードし、観客や他の審判と共有。現在のテキストシェア機能を補完する。

### 10. 音声フィードバック

- ショットクロック残り5秒: ビープ音
- ミス・バースト・勝利: 効果音（Web Audio API、ミュートトグル付き）

## 設計上の重要な観察

現在のアプリは **カジュアルプレイ向けとして非常に完成度が高い**。大会利用に向けた最大のギャップは以下の2点:

1. **ゲームを超えた管理層の欠如** — マッチ（Best-of-3）・シリーズ・統計
2. **審判機能の欠如** — ショットクロック・ファウル記録

スコアシートが既に実装されているのは大きな強みで、印刷対応を加えれば公式記録として即使用できる。マッチ管理機能を追加すると、公式大会の 1 試合分（Best-of-3）をこのアプリ 1 本で完結して運営できるようになる。
